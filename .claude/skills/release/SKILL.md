# /release — the mcpbolt release pipeline

This skill ships changes across **three surfaces**:

| Surface | Lives in | Release artifact |
| --- | --- | --- |
| **Mac app** | `mac-app/` | `mac-v<X.Y.Z>` GitHub tag + release + zip asset + Homebrew cask bump |
| **npm CLI** | `src/` | `cli-v<X.Y.Z>` GitHub tag + `npm publish` |
| **Landing**  | `landing/` | No tag. Vercel auto-deploys every push to `main`. |

The landing requires no release action — pushing to `main` is enough. This skill only orchestrates the mac app and the CLI.

## Repo map (memorize this — don't re-discover)

- Main repo: `github.com/vishmathpati/mcpbolt` (remote `origin`)
- Homebrew tap: `github.com/vishmathpati/homebrew-mcpbolt`, checked out on disk at `/opt/homebrew/Library/Taps/vishmathpati/homebrew-mcpbolt`
- Cask file: `/opt/homebrew/Library/Taps/vishmathpati/homebrew-mcpbolt/Casks/mcpboltbar.rb`
- npm package: `mcpbolt` (public, MIT). `package.json` at repo root.
- Mac app Info.plist is generated by `mac-app/build-app.sh`. Version string is in that script (look for `CFBundleShortVersionString` and `CFBundleVersion`).
- Mac app bundle id: `com.mcpbolt.MCPBoltBar`. Min macOS: 14 (Sonoma). Ad-hoc signed, not notarized.

## Preflight (always run first)

```bash
cd "$(git rev-parse --show-toplevel)"
git status -sb
gh auth status 2>&1 | head -3
brew --version | head -1
bun --version
```

Then confirm nothing is broken:

```bash
# working tree must be clean OR the user must explicitly say "include uncommitted changes"
git status --porcelain
# must be on main (or explicitly a release branch)
git branch --show-current
# local main must match origin/main
git fetch origin && git log HEAD..origin/main --oneline
```

If there are divergent changes, stop and surface them.

## Decide which surfaces to ship

Look at what's changed since the last release tag of each type.

```bash
# Last mac release
LAST_MAC=$(git tag -l 'mac-v*' --sort=-v:refname | head -1)
echo "Last mac: $LAST_MAC"
git diff --name-only "$LAST_MAC"..HEAD -- mac-app/ | head

# Last CLI release (tagged cli-v* OR simply compare src/ to published npm version)
LAST_CLI=$(git tag -l 'cli-v*' --sort=-v:refname | head -1)
if [ -z "$LAST_CLI" ]; then LAST_CLI="$LAST_MAC"; fi
echo "Last CLI: $LAST_CLI (or fall back to mac tag if no cli tag exists)"
git diff --name-only "$LAST_CLI"..HEAD -- src/ package.json | head
```

**Rule:** if a surface has no changes, **do not release it**. Tell the user, offer to skip.

Ask the user to pick the bump level (patch / minor / major) for each surface that changed. Default to patch unless they say otherwise.

Use `AskUserQuestion` (or just ask in chat) — never guess major vs minor without confirmation.

## Mac app release flow

### 1. Bump version in `mac-app/build-app.sh`

Two strings to edit — both `CFBundleShortVersionString` and `CFBundleVersion`.

```bash
NEW_VERSION="0.2.1"   # from the user
cd mac-app
# The build script has the version twice in its heredoc Info.plist
# Use Edit tool with replace_all=true on the old version string
```

Verify with `grep "CFBundleShortVersionString" -A1 build-app.sh`.

### 2. Build release + zip

```bash
cd mac-app
bash build-app.sh release
rm -f MCPBoltBar.zip
ditto -c -k --sequesterRsrc --keepParent MCPBoltBar.app MCPBoltBar.zip
ls -lh MCPBoltBar.zip
# Sanity check the built version matches
defaults read "$PWD/MCPBoltBar.app/Contents/Info" CFBundleShortVersionString
```

The version printed must equal `$NEW_VERSION`.

### 3. Commit version bump + tag

```bash
cd "$(git rev-parse --show-toplevel)"
git add mac-app/build-app.sh
git commit -m "chore(mac): bump to $NEW_VERSION"
git tag -a "mac-v$NEW_VERSION" -m "MCPBoltBar $NEW_VERSION"
git push origin main
git push origin "mac-v$NEW_VERSION"
```

### 4. Cut the GitHub release with the zip

Draft release notes first. They should cover only the mac-app changes since the last `mac-v*` tag. Use `git log "$LAST_MAC"..HEAD -- mac-app/ --oneline` to seed them.

```bash
gh release create "mac-v$NEW_VERSION" \
  --title "MCPBoltBar $NEW_VERSION" \
  --notes-file /tmp/mac-release-notes.md \
  mac-app/MCPBoltBar.zip
```

Show the user the returned release URL.

### 5. Bump the Homebrew cask

```bash
# Compute SHA256 of the uploaded zip (same file we just zipped)
SHA=$(shasum -a 256 mac-app/MCPBoltBar.zip | awk '{print $1}')
echo "SHA: $SHA"

CASK=/opt/homebrew/Library/Taps/vishmathpati/homebrew-mcpbolt/Casks/mcpboltbar.rb

# Edit version + sha256 in the cask. Do NOT touch macOS min unless build-app.sh
# changed LSMinimumSystemVersion. As of 0.2.0: Sonoma (14).

cd /opt/homebrew/Library/Taps/vishmathpati/homebrew-mcpbolt
git pull --ff-only origin main
# apply edits via Edit tool on $CASK:
#   version "OLD"   -> version "$NEW_VERSION"
#   sha256 "OLD..." -> sha256 "$SHA"

# Lint + audit BEFORE pushing
brew style --cask vishmathpati/mcpbolt/mcpboltbar
brew audit --cask --online vishmathpati/mcpbolt/mcpboltbar

git add Casks/mcpboltbar.rb
git commit -m "Bump mcpboltbar to $NEW_VERSION"
git push origin main
```

If you hit a push reject because someone else updated the cask (happened during 0.2.0 — a 0.1.5 bump was in flight), `git pull --rebase origin main`, resolve the conflict by keeping **your** newer version and sha, then `git rebase --continue` and push.

### 6. Verify the cask upgrades cleanly

```bash
brew update
brew upgrade --cask vishmathpati/mcpbolt/mcpboltbar
defaults read /Applications/MCPBoltBar.app/Contents/Info CFBundleShortVersionString
# must print $NEW_VERSION
```

Tell the user "relaunch MCPBoltBar.app, the update banner should be gone."

## npm CLI release flow

### 1. Bump version in `package.json`

```bash
NEW_CLI_VERSION="0.3.2"   # from the user — this is the NPM version
# Edit package.json "version" field
```

### 2. Typecheck + build

```bash
bun run typecheck
bun run build
ls -lh dist/cli/index.js
```

If typecheck fails, **stop and surface the error**. Do not publish broken code.

### 3. Publish

```bash
# Make sure you're logged in
npm whoami
# The package has prepublishOnly -> bun run build, so this will rebuild
npm publish
```

If `npm publish` errors on 2FA, prompt the user for the OTP and retry with `npm publish --otp=<code>`.

### 4. Commit + tag + push

```bash
cd "$(git rev-parse --show-toplevel)"
git add package.json
git commit -m "chore(cli): bump to $NEW_CLI_VERSION"
git tag -a "cli-v$NEW_CLI_VERSION" -m "mcpbolt CLI $NEW_CLI_VERSION"
git push origin main
git push origin "cli-v$NEW_CLI_VERSION"
```

### 5. (Optional) cut a GitHub release for the CLI

Small notes are fine. Users find it on npmjs.com anyway.

```bash
gh release create "cli-v$NEW_CLI_VERSION" \
  --title "mcpbolt CLI $NEW_CLI_VERSION" \
  --notes "See npmjs.com/package/mcpbolt for details."
```

## Landing

Landing deploys automatically on push to `main`. Nothing to release. If the user asks to "ship the landing," just confirm the latest commit is on `origin/main` and Vercel will handle it. If they want to force a promotion or inspect the deploy, point them at the Vercel dashboard — this skill does not touch Vercel via API.

If you ever bump landing deps for security (like the 15.0.3 → 16.2.4 fix), commit and push — Vercel will redeploy.

## Known gotchas (read before doing anything)

1. **Cask name is `mcpboltbar`, not `mcpbolt`.** The landing and the brew command must use `vishmathpati/mcpbolt/mcpboltbar` (the three-part `user/tap/cask` form).
2. **`mac-app/build-app.sh` has the version twice.** Miss one and `CFBundleVersion` drifts from `CFBundleShortVersionString`. Always `replace_all`.
3. **Ad-hoc signing means Gatekeeper will fight users.** The cask's `postflight` runs `xattr -cr` to strip quarantine. Do not remove that stanza.
4. **Homebrew cask min macOS must match `LSMinimumSystemVersion` in Info.plist.** `brew audit` will yell if they disagree. If you bump the Info.plist min, bump the cask `depends_on macos:` to match.
5. **Don't force-push the tap.** Other commits may be in flight. Always `git pull --rebase origin main` before pushing the cask.
6. **`git add -A` is banned.** Always add specific files. The repo has `.mcp.json.bak`, `node_modules/`, and a landing `.next/` that must never be committed.
7. **`npm publish` builds via `prepublishOnly`.** If the build is broken, publish will fail noisily — good. Don't use `--ignore-scripts`.
8. **Never publish if typecheck fails.** The CLI is small but it's public, and rollbacks on npm require a new version.

## Status report

When done, report:

```
RELEASED:
  - mac-v<X.Y.Z>  → https://github.com/vishmathpati/mcpbolt/releases/tag/mac-v<X.Y.Z>
  - Homebrew cask → bumped to <X.Y.Z>
  - (if applicable) npm mcpbolt@<X.Y.Z>  → https://www.npmjs.com/package/mcpbolt

NEXT:
  - User should relaunch MCPBoltBar.app to clear the update banner.
  - Verify Vercel redeploy if landing was part of this push.
```

Use the **DONE**, **DONE_WITH_CONCERNS**, **BLOCKED**, or **NEEDS_CONTEXT** status pattern at the end.

## Helper

`.claude/scripts/release-checks.sh` is a one-shot preflight + diff script. Run it at the start of every `/release` invocation:

```bash
bash .claude/scripts/release-checks.sh
```

It prints what changed per surface and surfaces anything that would block a release.
